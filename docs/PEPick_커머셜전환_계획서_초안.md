# PEPick 커머셜 전환 계획서 (초안 v0.1)

작성일: 2026-02-22  
대상: 근쌤 / PEPick 프로젝트

---

## 1) 문서 목적

이 문서는 PEPick을 **개인용 수업 도구 → 상업 서비스**로 전환하기 위한 1차 초안입니다.
핵심은 아래 4가지입니다.

1. 시장 조사(문제/수요/타겟)
2. 경쟁 서비스 비교 및 SWOT
3. 수익모델/결제모델 설계
4. 결제를 위해 필요한 코드/아키텍처 리팩토링 계획

---

## 2) 현재 PEPick 진단 (as-is)

### 제품 포지션

- 초등 체육 수업 맥락에 특화된 랜덤 운영 도구
  - 술래뽑기
  - 모둠뽑기
  - 학급/학생 관리
  - 타이머/휘슬/배지
- 교사 실사용 시나리오 중심 UX (카드 기반 학생 선택, 모달 흐름)

### 기술 포지션

- Vanilla JS + Vite + Firebase Auth/Firestore
- local-first + cloud sync 구조
- 모듈화가 상당히 진행되어 확장 토대는 있음

### 현재 강점 요약

- **체육수업 도메인 적합성** 높음 (범용 앱 대비 차별점)
- **현장 동선 친화 UX** (빠른 조작, 수업 중 즉시 사용)
- **기능-코드 매핑 명확** (tag/game/group/class 모듈 분리)

### 현재 한계 요약

- 상업 전환 관점의 기능(구독, 권한, 플랜 제한, 결제, 팀/학교 계약) 부재
- 데이터 계층은 있으나, **상용 SaaS 표준 객체(tenant/subscription/invoice/entitlement)** 부재
- 분석 지표(리텐션, 사용량, 전환율) 추적 기반 부족

## 3) 시장 조사 (1차 데스크 리서치)

## 3-1. 고객 문제 (Jobs-to-be-done)

교사가 실제로 원하는 것은 아래 4가지로 요약됩니다.

1. 수업 운영 시간 단축 (뽑기/팀편성/전환 시간)
2. 공정성 확보 (무작위/중복방지/이력)
3. 참여/동기 유도 (배지/게임화)
4. 수업 후 기록 정리 최소화

체육 교과는 특히 "즉시성"이 중요해서, 기능 풍부함보다 **한 번에 빠르게 작동하는 흐름**이 구매 요인입니다.

## 3-2. 타겟 세그먼트 제안

### 1차 타겟 (PMF 우선)

- 초등 체육 전담/담임 교사 (국내)
- 방과후 체육/스포츠 강사

### 2차 타겟

- 중등 체육 교사
- 국제학교/학원형 수업 운영자

### 3차 타겟 (B2B2C)

- 학교 단위 도입(부장/관리자 계약)
- 교육청 연수/교사 커뮤니티 제휴

## 3-3. 시장성 가설

- 범용 학급관리 앱은 많지만, **체육 활동 운영 특화 SaaS**는 상대적으로 희소
- 교사 개인 결제(B2C) + 학교 단체 계약(B2B) 혼합 전략이 유효
- 국내는 카드 + 간편결제(토스/네이버/카카오) 필수

## 4) 경쟁 서비스 비교 (요약)

> 아래는 공개 페이지 기반 데스크 리서치 1차 메모입니다.

### A. ClassDojo

- 강점: 대규모 사용자 기반, 학부모 커뮤니케이션, 포인트/SEL, Plus 구독 모델
- 한계: 체육 수업 운영 특화도는 낮음
- 시사점: PEPick은 "학부모 커뮤니케이션"보다 "수업 현장 운영 속도"에 집중

### B. TeacherKit

- 강점: 출결/성적/클래스 관리 범용성
- 한계: 체육 수업의 랜덤 게임 운영 UX는 약함
- 시사점: PEPick은 범용 LMS/학급관리와 다른 포지션(활동 운영 도구)

### C. iDoceo

- 강점: 교사용 올인원(성적/플래너/오프라인), 1회 구매형
- 한계: 학습곡선, 체육 활동 즉시 운영 특화 약함
- 시사점: PEPick은 "가볍고 즉시 사용"에서 우위 가능

### D. Wheel of Names류 랜덤툴

- 강점: 단순 랜덤 추첨, 사용 진입장벽 낮음
- 한계: 학급 데이터/모둠 고정/배지/수업 맥락 결여
- 시사점: PEPick은 단순 랜덤이 아니라 "수업 운영 시스템"으로 포지셔닝

## 5) SWOT 분석

## Strengths (강점)

1. **도메인 핏**: 체육수업 실전 문제를 정확히 겨냥
2. **UX 속도**: 수업 중 터치 수 최소화 구조
3. **모듈화 구조**: 상용 기능 추가를 위한 기술 토대 존재
4. **PWA+모바일 사용성**: 현장에서 설치형처럼 사용 가능

## Weaknesses (약점)

1. 결제/구독/권한 관리 부재
2. 관리자 기능(학교 단위 계정, 라이선스 배포) 부재
3. 데이터 인사이트(대시보드/리포트) 약함
4. 브랜딩/온보딩/상품 패키징 미정의

## Opportunities (기회)

1. AI·디지털 전환으로 교사용 유틸 도입 장벽 하락
2. 교사 커뮤니티 기반 바이럴 가능성
3. 체육 외 활동형 수업으로 확장 가능
4. B2C→B2B 업셀(학교 단위 계약) 여지

## Threats (위협)

1. 범용 에듀툴의 기능 흡수 가능성
2. 무료 랜덤툴과 가격 저항
3. 결제/개인정보/아동 데이터 관련 규제 이슈
4. 1인 개발 운영 시 SLA/지원 체계 부담

---

## 6) 제품 전략 제안 (포지셔닝)

## 포지셔닝 문장

**“PEPick은 체육 수업을 위한 실전 운영 OS”**

- 랜덤 추첨 도구가 아니라,
- "팀 구성 → 진행 → 기록/동기부여"를 하나로 묶는 수업 운영 플랫폼

## 핵심 차별화 축

1. 체육 활동 시나리오 프리셋
2. 모둠 고정/순환/균형화 로직
3. 배지·성장 시각화
4. 수업 진행 중 3터치 이내 완료 UX

---

## 7) 수익 모델 / 가격 모델 (초안)

## 7-1. 플랜 구조

### Free

- 기본 술래/모둠 뽑기
- 학급 1~2개 제한
- 기본 배지/기록

### Pro (개인 교사)

- 무제한 학급/학생
- 고급 모둠 로직(고정, 균형, 히스토리 기반)
- 배지/성장 리포트
- 템플릿/프리셋 저장
- 월/연 결제

### School (기관)

- 다교사 계정 관리
- 라이선스 일괄 배포
- 관리자 대시보드(사용량/활동)
- CS 우선 지원

## 7-2. 가격 방향 (가설)

- Pro: 월 5,900~9,900원 / 연 59,000~89,000원
- School: 좌석당 연 과금 또는 학교 단위 패키지

(실제 가격은 A/B 테스트 필요)

## 7-3. 결제 수단 우선순위 (국내)

1. 카드 정기결제
2. 토스페이/카카오페이/네이버페이
3. 계좌이체(학교 계약용)

## 8) 결제/수익화를 위한 코드 리팩토링 계획

## 8-1. 아키텍처 목표

기존 `users/classes/students` 구조 위에 아래 상용 객체를 추가:

- `tenants` (개인/학교 조직)
- `memberships` (사용자-조직 연결)
- `plans` (상품)
- `subscriptions` (구독 상태)
- `entitlements` (기능 권한)
- `billing_events` (결제 웹훅 로그)

## 8-2. 권장 데이터 모델 (예시)

```text
tenants/{tenantId}
  - type: personal | school
  - ownerUserId
  - planId
  - status

memberships/{membershipId}
  - tenantId
  - userId
  - role: owner | admin | teacher

subscriptions/{subscriptionId}
  - tenantId
  - provider: toss | stripe
  - status: active | past_due | canceled | trialing
  - currentPeriodEnd

entitlements/{tenantId}
  - maxClasses
  - advancedGrouping: true/false
  - analytics: true/false
  - teamSeats
```

## 8-3. 기능 게이트(Feature Flag) 도입

- 현재 UI/로직이 모든 기능 오픈 상태이므로,
- `EntitlementService`를 둬서 기능 접근을 중앙 통제

예:

- `canUseAdvancedGrouping()`
- `canCreateMoreClasses(currentCount)`
- `canViewAnalytics()`

## 8-4. 리팩토링 우선순위

### Phase A (2~3주): 결제 전 준비

1. 사용자/조직(tenant) 모델 도입
2. Entitlement 레이어 추가
3. 이벤트 로깅(가입, 핵심 기능 사용, 제한 초과 시도)

### Phase B (2~4주): 결제 통합

1. 결제 Provider 연동(우선 1개)
2. 결제 웹훅 처리 서버(Cloud Functions/Server)
3. 구독 상태 동기화 및 만료 처리

### Phase C (2주): 상용 UX

1. 플랜 페이지/업그레이드 모달
2. 무료 제한 안내 UX
3. 인보이스/결제 내역 화면

## 8-5. 코드 레벨 작업 항목

1. `js/shared/store.js` 확장
   - tenant/subscription/entitlement 상태 관리

2. `js/firestore-sync.js` 분리
   - billing sync 로직 분리 (`billing-sync.js`)
   - 실시간 구독 상태 반영

3. 라우트 보호
   - 유료 기능 접근 시 `PaywallModal` 노출

4. 관리자 페이지(신규)
   - `/billing`, `/plan`, `/team-admin` 뷰 추가

5. 감사 로그
   - 결제 이벤트/권한 변경 이벤트 로그 저장
